# 产品需求文档 (PRD)：ESP32 + MCP over MQTT 情感陪伴智能体

**所属系列**：从0到1打造“情感陪伴智能体”系列的第4篇 **文档状态**：已确认（V1.1）

------

## 1. 项目概述

### 1.1 项目核心目标

将 ESP32 开发板打造为一个具备“听、说”能力的 AI 情感陪伴设备。通过语音交互（用户输入语音 -> 云端 AI 理解情感并生成关怀回复 -> 设备播放语音），实现完整的智能硬件语音闭环，最终呈现一个“懂用户心情”的桌面小硬件伙伴。

### 1.2 产品优势

- **轻设备重云端**：ESP32 仅负责音频 I/O，核心算力和大模型处理全量部署在云端，兼顾低成本与高性能。
- **情感化交互**：通过多模态大模型精准识别用户语气，提供具备情绪价值的回复。
- **高实时性通信**：采用 MQTT 协议（EMQX），极大地优化了物联网设备的响应延迟。

------

## 2. 系统整体架构

整个系统分为硬件端、通信中间件、云端服务及移动端监控控制体系。

- **硬件端 (ESP32)**：负责 VAD 能量检测、音频采集（录音）、Base64 解码与 MP3 播放。
- **消息中间件 (EMQX)**：作为通信桥梁，利用 Webhook 将设备音频数据转发至后端，并将后端音频指令下发至设备端。
- **云端大脑 (Python / FastAPI)**：负责音频流的过滤、大模型（通义千问）的调度（ASR + 情感理解 + TTS）、音频格式转换（WAV 转 32kbps MP3）。
- **移动端 (Android)**：作为伴生 App 提供日志监控、远程语音发送、设备端静音控制及历史聊天记录沉淀。

## 3. 功能需求详细说明

### 3.1 硬件与边缘计算层 (ESP32)

- **引脚配置**：
  - **I2S 麦克风（录音）**：BCLK=7、LRCL=8、DOUT=9
  - **I2S 扬声器（播放）**：BCLK=2、LRCL=1、DOUT=42
  - *注：采用两套独立的 I2S 接口以避免读写冲突。*
- **采样参数**：8000Hz、16bit、单声道。固定单次录制时长为 3 秒。
- **交互与状态机管理**：
  - **VAD 触发**：麦克风持续检测语音能量，达到设定阈值且持续一定时间后触发录音。
  - **视觉反馈**：音频数据上传后、等待云端响应的空闲期内，板载 LED 需处于闪烁状态，以缓解用户等待焦虑。
  - **防回声/冷却机制**：播放结束后，必须清空音频缓冲区，并进入 3 秒的强制冷却期，此期间麦克风屏蔽任何能量输入。
  - **断线重连**：内置 MQTT 异常断开后的自动重连机制。

### 3.2 后端服务与 AI 链路 (Python + FastAPI)

- **通信接口**：提供 `/process_audio` 接收 EMQX Webhook 自动转发的 POST 请求。
- **前置过滤机制**：接收到音频后，首先进行静音/纯噪音检测。若判断为无效输入或纯环境噪音，直接丢弃，不调用大模型 API。
- **大模型处理 (qwen-omni-turbo)**：
  - 融合 ASR（语音识别）、情感分析及对话生成。
  - **Prompt 限制**：要求模型输出的回复必须简短精炼，不超过 20 个字。
  - 调用 TTS 服务合成语音（指定音色：Cherry）。
- **音频处理与下发**：
  - 引入 `lameenc` 将 TTS 生成的音频转换为低码率（32kbps）的 MP3 格式。
  - 将 MP3 数据进行 Base64 编码，通过 MQTT 协议下发至目标主题。
- **异常兜底机制**：若请求通义千问 API 发生超时或转换失败，服务需立刻向 ESP32 下发预置的异常提示音（例如：“抱歉，我刚刚走神了”），防止硬件无限期死等。

### 3.3 MQTT 主题规范

- **上传录音主题**：`emqx/esp32/audio`
- **下发播放主题**：`emqx/esp32/playaudio`

### 3.4 Android 移动端体系

- **技术选型更新**：全面弃用老旧的 `org.eclipse.paho.android.service`，改用适配高版本 Android 系统的现代 MQTT 客户端方案（如基于 Kotlin Coroutines 的 HiveMQ MQTT Client）。
- **核心功能迭代**：
  - **V1（监控器）**：实现日志回显、ESP32 状态监控及简单的 MQTT 文本消息发送功能。
  - **V2（双端联动）**：集成 `SpeechRecognizer`，实现手机端语音录制与发送；在 UI 层新增 **“静音 ESP32”** 切换开关，解决同一空间内多设备外放导致的杂音冲突问题。
  - **V3（全功能伴侣）**：引入 Material Design、RecyclerView 及 Room 数据库，完成本地消息持久化存储与双端数据同步。

## 4. 开发里程碑与部署计划

### Phase 1: 硬件与通信打基底（第 1 周）

- **目标**：跑通 ESP32 采集、MQTT 传输与 Python 后端闭环。
- **部署**：**本地局域网部署**（EMQX 与 Python 服务运行于本地 Linux 等开发机）。
- **验收标准**：对着 ESP32 说话，3秒后能够顺利播放出 AI 的回复；LED 交互正常；断网自动重连功能生效。

### Phase 2: 移动端引入与监控闭环（第 2-3 周）

- **目标**：完成 Android App V1 版本的开发与联调。
- **验收标准**：App 能稳定连接本地局域网中的 EMQX，实时打印 ESP32 的上下行数据日志，且能通过 App 按钮下发指令使 ESP32 发声。

### Phase 3: 云端迁移与深度联动（第 4-5 周）

- **目标**：实现手机语音聊天，加入并发与冲突控制机制。
- **部署**：**云端服务器部署**（将后台服务与中间件迁移至公有云），脱离同一 Wi-Fi 限制。
- **验收标准**：App 端语音发送逻辑顺畅，且“静音 ESP32”开关能精准切断硬件端的回放链路。

### Phase 4: 产品化收尾（第 6-8 周）

- **目标**：完成 Android 端的 UI 升级与数据持久化。
- **验收标准**：获得一个视觉美观、支持离线查看历史聊天记录的 Android 情感陪伴终端应用。
