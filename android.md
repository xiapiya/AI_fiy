产品需求文档 (PRD) - Android 专项重构版
所属系列： 从 0 到 1 打造“情感陪伴智能体”系列
文档状态： 已确认 (V4.2 极简解耦版)
项目代号： xiapiya (Android Client)

1. 项目定位与目标
1.1 定位
作为 xiapiya 智能体的移动端数字分身，提供远程文本交互能力。本客户端与桌面硬件端在操作逻辑上完全解耦，但在逻辑层面通过云端实现“统一意识（Unified Context）”的跨端覆盖。

1.2 核心目标
极致减法：剔除所有非核心的硬件控制逻辑，专注于文本对话。

无缝感知：通过共享云端上下文，实现跨设备对话记忆的连贯性。

敏捷交付：简化鉴权与多模态输入，确保核心链路快速跑通。

2. 业务功能需求
2.1 交互界面 (UI/UX)
主界面：采用标准流式布局（Message Feed）。

左侧气泡：显示 AI（Qwen-VL/Omni）生成的回复文本。

右侧气泡：显示主人输入的文本内容。

输入组件：底部单行/多行文本输入框，配合物理/虚拟发送按键。

视觉风格：延续 V4.1 的“极简几何风”，保持界面纯净，无冗余动画与功能按钮。

2.2 核心业务逻辑
独立对话模式：Android 端的对话请求直接上行至云端 FastAPI 路由，不触发桌面硬件（ESP32-S3）的任何动作（如语音播报、灯光反馈）。

跨端记忆同步：

上行触发：Android 端发送请求时携带统一的 Session_ID。

记忆闭环：AI 在处理请求时，将自动调用包含 ESP32 历史对话在内的完整上下文缓冲区。

静态状态管理：客户端不实时监听硬件在线状态，仅负责自身与云端的通信链路。

3. 技术架构与通信规范
3.1 身份锚定 (Identity Mapping)
免鉴权设计：为加速迭代，取消账号登录系统。

硬编码 ID：在客户端源码级定义全局常量 CLIENT_SESSION_ID，须与 ESP32-S3 固件中的 DEVICE_ID 保持一致，以此作为云端索引记忆的唯一凭据。

3.2 通信链路
下行流式传输：建议采用 SSE (Server-Sent Events) 或 WebSocket，实现 AI 回复文本的逐字流式渲染，提升交互流畅感。

数据持久化：使用 Jetpack Room 数据库对本地对话历史进行持久化存储，支持离线查看历史记录。

4. 接口与数据契约
4.1 文本上报接口 (POST)
Endpoint: /api/v1/chat/text

Payload:

JSON
{
  "session_id": "xiapiya_master_01",
  "content": "主人输入的文本内容",
  "timestamp": 1713514200
}
4.2 记忆共享逻辑
云端中枢接收到 session_id 后，从 Redis/MySQL 中提取关联的历史对话列表，拼接后提交至 LLM 推理引擎。

5. 开发路线图 (重构版)
Phase 1：搭建 Kotlin 基础框架，集成 Room 数据库与 OkHttp 通信模块。

Phase 2：实现硬编码 ID 逻辑，跑通与 FastAPI 的基础文本收发测试。

Phase 3：优化 UI 渲染，实现流式文本显示，确保跨端上下文在云端正确闭环。

Phase 4：弱网兼容性测试及内存泄漏排查。
